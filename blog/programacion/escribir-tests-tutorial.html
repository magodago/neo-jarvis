Guía práctica para escribir tests usando inteligencia artificial

26 June 20267 min de lectura

Por qué tus tests no están haciendo su trabajo real

Si llevas más de un año programando, seguro que has heredado un repositorio con cientos de tests que siempre pasan en verde pero no cubren ni el treinta por ciento de los fluros reales de la aplicación. Esa falsa sensación de seguridad es más peligrosa que no tener tests. En 2026, con la madurez de herramientas como Jest 30, Vitest 3.2 y frameworks de testing end-to-end como Playwright 1.50, no hay excusa para seguir escribiendo pruebas que solo verifican el camino feliz. La clave está en pasar de tests que repiten la implementación a tests que validan el comportamiento esperado, incluyendo errores, estados límite y condiciones de carrera.

Un test mal escrito no es mejor que ningún test. Puede darte una cobertura del 95% en las métricas de SonarQube, pero si nunca lanzas una excepción real o nunca pruebas con datos corruptos, ese porcentaje es un espejismo. Por eso, antes de escribir una sola línea de test, necesitas definir qué es lo que realmente quieres proteger: la lógica de negocio, la integración con servicios externos o la experiencia del usuario final.

**PROMPT:** "Actúa como un ingeniero de calidad senior con experiencia en [lenguaje/framework] y [nombre del proyecto]. Quiero que generes una lista de al menos diez casos de prueba (unitarios y de integración) para la función [nombre de la función/módulo] que se encuentra en [ruta del archivo]. Los casos deben cubrir: entrada válida, entrada inválida, valores límite, errores esperados, estado vacío, concurrencia si aplica, y un escenario de fallo de dependencia externa. Para cada caso, indica el nombre descriptivo del test, el tipo (unitario/integración), las condiciones previas y el resultado esperado."

---

Cómo estructurar un test que sobreviva a los refactors

La razón principal por la que los tests se vuelven frágiles es porque están acoplados a los detalles de implementación. Un test que llama a un método privado, que mockea una función interna que cambia de nombre en la siguiente versión, o que verifica el orden exacto de las llamadas a base de datos, se rompe sin que la funcionalidad haya cambiado. En 2026, con arquitecturas orientadas a eventos y microservicios, esto es especialmente crítico.

La solución es escribir tests desde la perspectiva del contrato público: lo que entra y lo que sale, no cómo se procesa internamente. Por ejemplo, si tienes un servicio de envío de notificaciones, no debes testear que se llamó a `enviarCorreoInterno()`, sino que, dado un usuario y un mensaje, el sistema persiste la notificación y devuelve un código 201. Esto permite cambiar la implementación interna sin tocar los tests.

Un patrón que funciona muy bien es Given-When-Then para tests unitarios, y Setup-Execute-Verify para tests de integración. Sobre todo, evita los datos mágicos: cada valor en un test debe tener una razón semántica. Por ejemplo, usar `usuarioConSaldoCero` en lugar de un número aleatorio.

**PROMPT:** "Eres un desarrollador experto en testing con [framework elegido, ej: pytest + pytest-mock o Jest + nock]. Necesito que me ayudes a refactorizar los tests del módulo [nombre del módulo] que actualmente verifican implementación interna (stubs a métodos privados, llamadas a funciones helper). Propón una versión alternativa de los tests que solo validen el comportamiento observable: entradas públicas, salidas, efectos secundarios documentados (como llamadas a API externas o escritura en base de datos). Incluye un ejemplo concreto de cómo reemplazar un mock de un método privado por un stub de la dependencia externa real."

---

Tests de integración que no necesitan un entorno de producción

Muchos equipos caen en la trampa de creer que los tests de integración requieren infraestructura real: bases de datos dedicadas, colas de mensajes, servicios externos. En 2026, herramientas como Testcontainers con Java o Python, y sus equivalentes para Node con Testcontainers Node, permiten levantar contenedores desechables para cada suite de pruebas. Pero aún así, muchos desarrolladores siguen mockeando todo y perdiendo la validación real.

Un test de integración debe verificar que dos o más componentes se comunican correctamente: por ejemplo, que un controlador REST recibe un JSON, lo transforma correctamente, lo persiste en PostgreSQL y devuelve la respuesta esperada. No necesita ser una réplica exacta de producción, pero sí debe usar el mismo driver de base de datos, el mismo serializador y la misma lógica de errores. Usar un contenedor de PostgreSQL efímero te da la tranquilidad de que la query funciona, no solo que el mock de la base de datos devuelve un objeto.

Un error habitual es incluir en estos tests datos demasiado grandes o conexiones lentas, lo que los vuelve lentos. La solución es limitar el alcance: prueba un flujo vertical corto (controlador -> servicio -> repositorio) sin incluir la autenticación externa ni el envío de emails.

**PROMPT:** "Genera un plan detallado para escribir tests de integración automatizados para la API REST de [nombre del proyecto] usando [herramienta específica: Supertest + Testcontainers para Node, o Flask-Testing + pytest-django + testcontainers para Python]. El plan debe incluir: 1) Configuración del contenedor de base de datos con datos semilla mínimos. 2) Lista de tres endpoints críticos a probar (uno de creación, uno de consulta con filtros, uno de borrado con validación de permisos). 3) Cómo aseverar el estado de la base de datos después de cada operación. 4) Estrategia para limpiar datos entre tests sin afectar la velocidad. Adjunta un fragmento de código real para el test del endpoint POST /usuarios."

---

Errores que siguen cometiendo los equipos en 2026 (y cómo evitarlos)

A pesar de la madurez de la industria, hay patrones de testing que se repiten sin sentido. El más común es el *overmocking*: mockear cada objeto y función, incluso los que nunca fallan. Esto hace que el test solo pruebe el mock, no el código real. La regla de oro: mockea solo lo que está fuera de tu control (APIs de terceros, servicios de autenticación, el reloj del sistema). No mockees tu propia base de datos ni tu propio sistema de archivos si puedes evitarlo.

Otro error es probar demasiado lo que no importa. Escribir un test unitario para un getter o para un método que solo delega sin lógica no añade valor. Cada test tiene un coste de mantenimiento. En vez de perseguir el 100% de cobertura (que es una métrica inservible sin calidad), prioriza las rutas críticas: flujos de pago, registro de usuarios, procesamiento de archivos y cualquier lógica que involucre condiciones complejas.

El tercer error es no mantener los tests como parte del código. Si un test falla y nadie lo revisa, o si se ignora porque «es un falso positivo», la cultura de calidad se va al piso. En 2026, con CI/CD ultrarrápidos, cada test fallido debe detener el pipeline. Y si un test es inestable (flaky), hay que arreglarlo o eliminarlo, no marcarlo como skip.

**PROMPT:** "Eres un consultor técnico especializado en calidad de software. He identificado que mi equipo comete estos tres errores habituales en testing: [overmocking / probar getters / ignorar tests inestables]. Quiero un plan de acción concreto, paso a paso, para corregir cada uno en un plazo de dos semanas. Incluye: cómo revisar los tests existentes para encontrar overmocking, criterios para decidir qué tests eliminar o fusionar, y un proceso para identificar y estabilizar tests flaky mediante reintentos inteligentes o aislamiento de datos. Dame también un ejemplo de un test refactorizado que pase de mockear todo a usar un stub controlado con [nombre de librería, ej: WireMock o MockServer]."

---

Conclusión: el test que escribes hoy te ahorrará una madrugada mañana

Escribir tests no es un trámite que hay que cumplir para alcanzar un porcentaje en la herramienta de CI. Es una inversión en la capacidad de cambiar el código con confianza. Un test bien diseñado se convierte en la documentación viva del sistema, actualizada cada vez que pasa en verde. Los prompts que te he compartido no son una receta mágica, sino una guía para que dejes de delegar la calidad a la intuición y empieces a usar la inteligencia artificial como un aliado para estructurar casos de prueba robustos.

Empieza por uno de tus módulos más críticos: aplica el prompt para listar casos reales, refactoriza los tests frágiles y configura un par de tests de integración con contenedores. En una semana notarás la diferencia cuando un cambio que antes rompía todo ahora solo necesite ajustar una línea. La disciplina de probar bien no es opcional; es lo que separa un producto profesional de un prototipo. Así que abre tu terminal, escribe `npm test` y asegúrate de que cada resultado rojo te enseñe algo nuevo sobre tu código.
